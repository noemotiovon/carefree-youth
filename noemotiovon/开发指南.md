# 提 ISSUE

这一步并非必需。它的主要作用是让社区成员了解你正在进行的工作。通常只有在**较大规模的功能开发**中才需要先提交 Issue（目前给学生布置的任务大多数不属于大型功能，因此可以选择性省略。当然，如果愿意，也可以简要写一下计划实现的功能）。

**建议 ISSUE 命名**：

`[CANN] xxxxxxxxxx`

以下是推荐的 RFC 模板示例：

```
# 概述 (Summary)
简要描述你想提出的改动或新增的功能。

# 背景 (Motivation)
说明提出此改动的原因，包括当前的问题、痛点或限制。

# 问题描述
具体说明使用场景，以及为何现有方案不能满足需求。

# 目标 (Goals)
列出本 RFC 希望达成的目标。
- 目标 1
- 目标 2

# 详细设计 (Detailed Design)
详细描述设计思路，包括实现方案、接口变更、数据结构、算法流程等。
- 方案 A
- 方案 B
```

# 提 PR

提交 PR 的目的，是将你在自己 fork 仓库中完成的修改请求合入 `llama.cpp` 主仓库的 `master` 分支。

**建议 PR 命名规范：**
 `CANN: xxxxxxxxxx`

以下为推荐的 PR 模板：

```
# 描述 (Description)
简要说明本次 PR 的内容和目的。
如有对应 Issue，请注明：相关问题/功能: (#issue)

# 测试 (Testing)
描述验证本次修改正确性的方法，包括测试步骤和测试结果。

# 备注 (Notes)
其他需要强调或讨论的事项。
```

# 交流流程

提交 PR 后，等待社区的 Maintainer 进行 review。你也可以通过 @hipudding 和 @noemotiovon 通知我们，我们会在空闲时进行审阅并留下反馈。

根据 review 意见修改代码后，只需将更新 push 到你 PR 所对应的分支即可，**无需重新提交新的 PR**。PR 会自动更新，因为它追踪的是你 fork 仓库中的对应分支到 `llama.cpp` 主仓库 `master` 分支的变更情况。
